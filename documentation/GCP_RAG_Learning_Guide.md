# GCP RAG Agent: Learning & Deployment Guide

This document provides a comprehensive overview of the Retrieval-Augmented Generation (RAG) agent built on Google Cloud Platform. It is designed for learning, revising, and understanding the end-to-end architecture.

---

## 🏗️ Architecture Overview

The project implements a **RAG (Retrieval-Augmented Generation)** pattern. Instead of relying solely on an LLM's pre-trained knowledge, we "augment" its prompt with specific, real-time data retrieved from a private corpus.

### The Flow:
1. **User Query:** User asks a question (e.g., "What is the PTO policy?").
2. **Retrieval:** The system searches a managed **Vertex AI RAG Corpus** for the most relevant text chunks.
3. **Augmentation:** The retrieved chunks are combined with the original user query.
4. **Generation:** The combined context is sent to **Gemini 1.5**, which generates a natural language answer based *only* on the provided data.

---

## 🛠️ GCP Services Used

### 1. Vertex AI RAG Engine
* **Purpose:** Acts as the "Brain" for data management.
* **Why we used it:** It provides a fully managed API to create a "Corpus," upload documents, and perform vector similarity searches without needing to manage a separate vector database (like Pinecone or FAISS) or handle complex embedding logic manually.
* **Key Learning:** It handles document storage, chunking, and vectorization behind the scenes.

### 2. Vertex AI Gemini (1.5 Pro/Flash)
* **Purpose:** The generative LLM.
* **Why we used it:** It is the state-of-the-art model on GCP. We used `gemini-1.5-pro-001` for generation because it has a large context window and strong reasoning capabilities.
* **Key Learning:** We integrated it with a `Tool` (VertexRagStore) so the model knows how to "fetch" data when it doesn't have the answer.

### 3. Google Cloud IAM & Project Management
* **Project ID:** `indigoifs`
* **Region:** `us-west1` (Used because `us-central1` frequently hits capacity limits for new RAG Engine projects).

## 🚀 Implementation Steps (What we did and Why)

### Step 1: Environment & Auth
* **Action:** Set up `gcloud` auth and a Python virtual environment.
* **Why:** To securely interact with GCP APIs and manage dependencies like `google-cloud-aiplatform` and `langchain`.

### Step 2: Data Preparation
* **Action:** Created `data/hr_policy.txt`.
* **Why:** To have a ground-truth document that the LLM was *not* trained on, allowing us to prove the RAG system works.

### Step 3: Creating the RAG Corpus (`create_rag_corpus.py`)
* **Action:** Initialized a `RagCorpus` in `us-west1`.
* **Why:** To define a logical "container" for our embeddings and documents.
* **Critical Learning:** We used `rag.upload_file` for local-to-cloud upload because `rag.import_files` requires a Google Cloud Storage bucket (which requires additional permissions).

### Step 4: Connecting the Agent (`qa_agent.py`)
* **Action:** Built a script that creates a `Tool.from_retrieval`.
* **Why:** This tells Gemini: "If you get a query, don't just guess. Use this specific RAG tool to find the facts first."
* **Key Logic:** `similarity_top_k=2` ensures the model gets the top 2 most relevant snippets of text from the policy.

## 📝 Troubleshooting & Lessons Learned

1. **Capacity Issues:** If a region like `us-central1` fails with "capacity limitation," switch to `us-west1` or `europe-west1`.
2. **Direct Uploads:** You don't always need a GCS bucket. For simple test agents, `rag.upload_file` is much faster and bypasses Storage Admin permission requirements.
3. **Model Availability:** If `gemini-1.5-flash` is missing in one region, you can explicitly point the `MODEL_ID` to a region where it is available (e.g., `us-central1`) while keeping your RAG data in another.
